## 1. Overview

PyArchInit's `us_table` exposes 7 columns that act as logical grouping dimensions for stratigraphic units (US):

| Column | Type | Domain |
|---|---|---|
| `area` | TEXT | sub-area within a site |
| `struttura` | VARCHAR(30) | physical structure: "basilica", "chiesa", "area sacrale", "ED01/FC01" |
| `attivita` | VARCHAR(4) | EM activity code |
| `settore` | TEXT | sector |
| `ambient` | TEXT | room / ambient |
| `saggio` | TEXT | sondage |
| `quad_par` | TEXT | square partition |

Today (after AI05 ships 5.4.0-alpha), the green "Esporta Extended Matrix" button produces a GraphML where the only logical grouping is the epoch swimlane (TableNode generated by s3dgraphy `EpochSwimlanesGenerator`). The 7 column-driven groupings live only as round-trip data keys (`pyarchinit.struttura` etc., from AI04) attached to each US — no visual grouping, no knowledge-graph edge.

AI06 adds:

1. A **GraphProjector path** that materializes one s3dgraphy `ActivityNodeGroup` per `(dimension, value)` pair, with a discriminator attribute `group_kind ∈ {area, struttura, attivita, settore, ambient, saggio, quad_par, adhoc}`, plus an `is_in_activity` edge from each member US.
2. A **post-export injector** in `graphml_writer.py` that re-renders these groups as yEd folder nodes (`yfiles.foldertype="group"`, `<y:GroupNode>` realizer with dashed grey border + fill `#F5F5F5` + label-on-top), nested inside the existing TableNode swimlane and positioned to span all epoch rows of their member US.
3. A **GroupStore** (CRUD analogous to AI05's ParadataStore) for ad-hoc groups not derivable from SQL, persisted to `groups_{sito_slug}.graphml` next to the SQLite DB.
4. **UI extensions:** a 7-checkbox "Group US by" panel in S3DGraphyExportDialog (Export tab) with auto-preselection of populated dimensions, plus a 4th tab "Groups" in the existing ParadataManagerDialog.
5. A **CLI extension** with `paradata add-group / list-groups / add-us-to-group / remove-group` sub-subcommands and an `--group-by struttura,attivita,...` flag on `export`.
6. A **round-trip flag** in the Import tab that lets the user write SQL UPDATEs reflecting group-membership changes made in yEd (default OFF for safety).

The grouping is **opt-in pure**: default behaviour (no checkbox, `groups=[]`) preserves the AI05/AC-2 byte-identical baseline unchanged.

## 2. Decisions (D1–D7)

Resolved during the brainstorming session 2026-05-08. Each decision pins behaviour testable in the acceptance criteria (§7).

### D1 — Scope of dimensions

**Choice: A — all 7 dimensions.** Even though three (`ambient`, `saggio`, `quad_par`) are typically empty on current real-world DBs, the design supports the full PyArchInit data model.

### D2 — UI selector

**Choice: B + D — checkboxes with auto-preselect.** A `QGroupBox "Group US by"` with one `QCheckBox` per dimension. On dialog open, `dimensions_with_data(db, sito)` runs once and pre-checks each dimension that has at least one non-empty value in `us_table`. The user can override before clicking Export. An additional `cb_grp_adhoc` checkbox handles ad-hoc groups from the GroupStore file.

### D3 — yEd rendering

**Choice: EM canonical template** (matches `~/Downloads/EM_demo_02.graphml`).

```
<node id="grp_<uuid>" yfiles.foldertype="group">
  <data key="dN">
    <y:ProxyAutoBoundsNode>
      <y:Realizers active="0">
        <y:GroupNode>
          <y:Geometry x=... y=... width=... height=.../>
          <y:Fill color="#F5F5F5" transparent="false"/>
          <y:BorderStyle color="#000000" type="dashed" width="1.0"/>
          <y:NodeLabel
              alignment="right" autoSizePolicy="node_width"
              backgroundColor="#EBEBEB" fontFamily="Dialog" fontSize="15"
              modelName="internal" modelPosition="t" verticalTextPosition="bottom">
            {group.name}
          </y:NodeLabel>
          <y:Shape type="roundrectangle"/>
          <y:State closed="false"/>
        </y:GroupNode>
      </y:Realizers>
    </y:ProxyAutoBoundsNode>
  </data>
  <graph id="grp_<uuid>:" edgedefault="directed">
    <!-- member US <node> elements re-parented here, original Geometry preserved -->
  </graph>
</node>
```

The group folder is a child of the TableNode (the epoch swimlane container). Its `<y:Geometry>` is computed as the bounding box of all member US nodes — so visually it spans every epoch row where any member sits, rendered as a dashed background.

### D4 — Edge semantics in the knowledge graph

**Choice: E — pragmatic now, migrate later.**

For all 7 dimensions, AI06 instantiates `s3dgraphy.nodes.group_node.ActivityNodeGroup` and adds `is_in_activity` edges (the only group-targeting edge in s3dgraphy 0.1.40 datamodel). A discriminator attribute `node.attributes["group_kind"]` distinguishes structural / locational / activity semantics within the file.

This is "tactically correct" because the s3dgraphy `ActivityNodeGroup` docstring defines an Activity as *"a logical group of actions held together for narrative and information ordering purposes"* — a basilica IS such a group. The lexical mismatch ("activity" vs "location") is documented and reversible.

The clean-alignment path (a new `LocationNodeGroup` class + `is_in_location` edge) is opened upstream as **issue #5** at `zalmoxes-laran/s3Dgraphy`. AI07 will migrate via a 1-line `if/else` in `_merge_groups` once the upstream change lands. Migration is non-breaking: the `group_kind` attribute is already preserved through round-trip.

### D5 — Round-trip semantics

**Choice: C — configurable via flag, default safe.**

The Import tab in `S3DGraphyExportDialog` gains a new `QCheckBox "Update SQL on import (struttura/area/attivita/settore/ambient/saggio/quad_par)"`, default unchecked. When unchecked: the importer reads group membership but does NOT issue SQL UPDATEs for the 7 SQL-derived columns (groups are decorative on import). When checked: each yEd-modified membership produces a `UPDATE us_table SET <group_kind>=<group.name> WHERE id_us=<row.id>` within an atomic transaction (AI04 contract — full rollback on any error).

Ad-hoc groups (`group_kind="adhoc"`) are NEVER written to SQL regardless of the flag. They only update the GroupStore membership file.

### D6 — Persistence of ad-hoc groups

**Choice: B — extend the dialog with a "Groups" tab + groups_*.graphml file.**

`gui/dialog_paradata_manager.py` (the AI05 dialog) gains a 4th tab: Groups. The tab provides:

- a `QTableWidget` listing existing groups (columns: name, group_kind, US count)
- form fields: name, kind dropdown (initially `["adhoc"]` only — SQL-derived kinds enter via round-trip, not user input), and a "Pick US members…" button that opens a secondary dialog with a multi-select picker of US in the current sito
- Add / Remove buttons that delegate to `GroupStore`

Persistence: `{db_dir}/groups_{sito_slug}.graphml`, where `sito_slug = re.sub(r'\W', '_', sito).lower()` (consistent with AI05's `paradata_*.graphml`). Atomic write via `.tmp` + `os.replace()`.

### D7 — Backward compatibility (AC-2 baseline)

**Choice: A — opt-in pure.** Default `groups=[]` preserves the existing AI05/AC-2 byte-identical export pipeline. No baseline regeneration needed. Stage 4e (`_inject_group_folders`) is a no-op when the snapshot is empty. The fixture `mini_volterra.sqlite` is exported with `groups=[]` in AC-2 tests.

## 3. Components

### 3.1 `GroupStore` (new — `modules/s3dgraphy/sync/group_store.py`)

Mirror of `ParadataStore` (AI05) for ad-hoc groups.

```python
class GroupStore:
    def __init__(self, db_path: Path, sito: str) -> None
    @property
    def file_path(self) -> Path  # {db_dir}/groups_{sito_slug}.graphml
    def exists(self) -> bool
    def read(self) -> 's3dgraphy.Graph'
    def write(self, graph) -> None
    def add_group(self, name: str, group_kind: str = "adhoc",
                  member_us_uuids: list[str] = None,
                  description: str = None) -> str  # returns uuid7
    def list_groups(self) -> list[dict]
    def remove_group(self, group_uuid: str) -> None
    def add_us_to_group(self, group_uuid: str, us_uuid: str) -> None
    def remove_us_from_group(self, group_uuid: str, us_uuid: str) -> None
    def list_members(self, group_uuid: str) -> list[str]
    def find(self, group_kind: str = None, **kwargs) -> list

class GroupStoreError(GraphSyncError): ...
class GroupReadError(GroupStoreError): ...
class GroupWriteError(GroupStoreError): ...
class GroupValidationError(GroupStoreError): ...
class GroupNotFoundError(GroupStoreError): ...
```

**Invariants (testable):**
- non-empty `name` required (else ValidationError)
- atomic crash-safe writes (mock `os.replace` raise → original untouched, `.tmp` cleaned)
- `read()` returns empty `Graph` when file absent (non-fatal)
- `add_us_to_group` is idempotent
- defensive type filter: `read()` returns only nodes with `group_kind` attribute (drops unrelated types)

### 3.2 `group_projector` (new — `modules/s3dgraphy/sync/group_projector.py`)

Pure-functional helpers; no class state.

```python
@dataclass(frozen=True)
class GroupSpec:
    group_uuid: str
    name: str
    group_kind: str  # one of {area, struttura, attivita, settore, ambient, saggio, quad_par, adhoc}
    member_us_uuids: list[str]
    description: str = ""

def build_groups_from_sql(
    db_path: Path, sito: str, dimensions: list[str]
) -> list[GroupSpec]: ...

def merge_adhoc_groups(
    sql_specs: list[GroupSpec], store: 'GroupStore'
) -> list[GroupSpec]: ...

def dimensions_with_data(
    db_path: Path, sito: str
) -> list[str]: ...
```

UUID generation: deterministic UUID5 from `(sito, group_kind, name)` so re-export with the same SQL state produces identical UUIDs (idempotent, AC-7).

### 3.3 `GraphProjector.populate_graph` extension

Single new kwarg + a private method.

```python
def populate_graph(
    self, db_path, sito, *,
    include_paradata: bool = True,    # AI05
    strict_schema: bool = True,       # AI05
    groups: list[str] = None,         # NEW (AI06 D7-A: None/empty = no grouping)
) -> 's3dgraphy.Graph': ...

def _merge_groups(self, graph, db_path, sito, dimensions): ...
```

`_merge_groups` builds `GroupSpec` instances via `group_projector.build_groups_from_sql`, optionally merges ad-hoc via `merge_adhoc_groups` (when `"adhoc"` is in `dimensions`), then materializes `ActivityNodeGroup` instances on `graph.nodes` plus `is_in_activity` edges from each member US.

### 3.4 `_inject_group_folders` (new — in `graphml_writer.py`)

Stage 4e, runs after the AI05 paradata injector.

```python
def _inject_group_folders(
    group_snapshot: list,           # ActivityNodeGroup pre-export snapshot
    members_map: dict[str, list[str]],  # group_uuid → [us_emid, ...]
    xml_path: Path,
) -> None: ...
```

Pre-export snapshot is captured BEFORE Stage 2 `_filter_by_site` (same lesson learned during AI05 H.4 smoke gate, when `_filter_by_site` was found to drop AuthorNode/LicenseNode/EmbargoNode). The snapshot pattern is consistent with `_inject_isolated_paradata_nodes` (AI05).

The post-processor parses the GraphML output via lxml, finds the TableNode wrapper, computes the bbox of each group's member US, creates a `<node yfiles.foldertype="group">` element with the EM canonical layout (D3), and re-parents the member US `<node>` elements from the TableNode-level `<graph>` to the new group's `<graph>` container. Original geometry of the US is preserved so they continue to render in the correct epoch swimlane row.

### 3.5 ParadataManagerDialog — 4th tab "Groups"

Extends `gui/dialog_paradata_manager.py`. Stays Qt5/Qt6 compatible via the `qgis.PyQt` abstraction (AI05 pattern).

The "Pick US members…" button opens a secondary `QDialog` with a `QListWidget` populated from `us_table` filtered by sito, multi-select enabled.

### 3.6 S3DGraphyExportDialog — checkboxes + import flag

Extends the existing dialog (`modules/s3dgraphy/s3dgraphy_dot_bridge.py`).

Export tab adds:
```python
self.gb_groups = QGroupBox("Group US by (optional)")
# 7 checkboxes (one per dimension) + 1 "ad-hoc" checkbox
# Auto-preselect via dimensions_with_data() on dialog open
```

Import tab adds:
```python
self.cb_sql_apply_groups = QCheckBox(
    "Update SQL on import (struttura/area/attivita/settore/ambient/saggio/quad_par)"
)
self.cb_sql_apply_groups.setChecked(False)  # D5-C default safe
```

The flag is passed to `GraphIngestor.populate_list(... sql_apply_groups=...)`.

### 3.7 CLI extensions (`scripts/s3dgraphy_sync.py`)

Four new sub-subcommands under `paradata`:

```
paradata add-group --db --sito --name [--kind adhoc] [--us-uuid ...]
paradata list-groups --db --sito
paradata add-us-to-group --db --sito --group-uuid --us-uuid
paradata remove-group --db --sito --uuid
```

Plus a flag on `export`:

```
export --db --sito --output --group-by struttura,attivita,adhoc
```

Exit codes: 0 success / 1 GraphSyncError (incl. GroupStoreError, ProjectionError) / 2 argparse error.

## 4. Data flow

### 4.1 Export with grouping (primary scenario)

```
1. User clicks "Esporta Extended Matrix" (US scheda green button)
2. S3DGraphyExportDialog._preselect_groups(db, sito)
     → group_projector.dimensions_with_data(db, sito)
     → cb_grp[d].setChecked(d in populated)
3. User adjusts checkboxes, clicks Export
4. bridge.s3dgraphy_to_graphml(...) builds groups arg
   groups = ["struttura", "attivita"]   # example
5. graphml_writer.export_graphml(db, mapping, out, groups=...)

   Stage 1: GraphProjector.populate_graph(
              ..., include_paradata=True, groups=["struttura","attivita"])
            ├ build_strat_layer (AI05 unchanged)
            ├ _merge_paradata (AI05 unchanged)
            └ _merge_groups (NEW)
                ├ build_groups_from_sql → [GroupSpec, GroupSpec, ...]
                ├ merge_adhoc_groups (no-op if "adhoc" not in groups)
                └ for each spec:
                    ActivityNodeGroup(node_id=spec.group_uuid,
                                      attributes={"group_kind": ...,
                                                  "sito": sito,
                                                  "pyarchinit.<kind>": name})
                    graph.add_node(group_node)
                    for us_uuid in spec.member_us_uuids:
                        graph.add_edge(...src=us_uuid,
                                       ...dst=group_uuid,
                                       ...edge_type="is_in_activity")

   Pre-export snapshot:
     _paradata_snapshot = [...]                       # AI05
     _group_snapshot    = [...all GroupSpec.materialized...]   # NEW
     _members_map       = {grp_uuid: [us_emid, ...]}

   Stage 2 _filter_by_site (AI05 unchanged; drops paradata + groups)
   Stage 3 GraphMLExporter.export(out)               (AI05 unchanged)
   Stage 4b/c/d (AI05 unchanged: visual / data-keys / paradata inject)
   Stage 4e _inject_group_folders(_group_snapshot, _members_map, out)  (NEW)
            parse out via lxml
            find TableNode <graph id="n0:">
            for each group in snapshot:
              members = list of <node> elements by EMID
              bbox = union of member geometries + margin
              create <node yfiles.foldertype="group" id="grp_{uuid}">
                with <y:GroupNode> realizer (dashed border, #F5F5F5 fill,
                                             label position=top, bg=#EBEBEB)
                with <graph id="grp_{uuid}:">
                  re-parent member <node> elements here
            tree.write(out, xml_declaration=True, pretty_print=True)
```

### 4.2 Round-trip: user moves US in yEd, re-imports

```
User opens Extended_Matrix.graphml in yEd, drags US 5 from "basilica"
folder to "chiesa" folder, saves.

User imports modified GraphML via Import tab.
Checkbox "Update SQL on import" decides behaviour:

  CHECKED:
    GraphIngestor.populate_list(graphml_path, sql_apply_groups=True)
      ├ parse GraphML via lxml
      ├ for each <node yfiles.foldertype="group">:
      │     read group_kind + name from data keys / NodeLabel
      │     read children US emids from <graph id="grp_X:">
      ├ for each child US emid:
      │     find SQL row by node_uuid
      │     IF group_kind in 7 SQL dimensions:
      │         queue UPDATE us_table SET <kind>=<name> WHERE id_us=<id>
      │     IF group_kind == "adhoc":
      │         queue GroupStore.add_us_to_group(group_uuid, us_uuid)
      ├ atomic transaction:
      │     all SQL UPDATEs in one BEGIN/COMMIT
      │     GroupStore.write atomic via .tmp + os.replace
      │     ROLLBACK both on any error → GraphSyncError
      └ return IngestResult(applied=N, updated=N, skipped=0)

  UNCHECKED (default safe):
    Same parsing, but only ad-hoc memberships are applied.
    SQL-derived movements are skipped silently.
    IngestResult.skipped reports the count of ignored UPDATEs.
```

### 4.3 Ad-hoc group creation via dialog

```
User clicks "Manage paradata" → dialog (4 tabs).
User clicks "Groups" tab, types name "restauri-2023", kind="adhoc".
User clicks "Pick US members..." → secondary dialog, multi-select 5 US, OK.
User clicks "Add group".
  → store.add_group(name="restauri-2023", group_kind="adhoc",
                     member_us_uuids=[u1,...,u5])
      └ GroupStore.write(graph_with_new_group)
          ├ _write_minimal_graphml (custom lxml serializer, AI05 pattern)
          ├ _embed_pyarchinit_data_keys (AI04 helper)
          └ os.replace(.tmp, groups_<slug>.graphml)
  → table refresh: "restauri-2023 | adhoc | 5"

User does Extended Matrix export with cb_grp_adhoc.checked=True
  → group + 5 edges materialized in output GraphML.
```

## 5. Error handling

### 5.1 Exception hierarchy

```
GraphSyncError                         (AI04 base — extends Exception)
├── ParadataStoreError                 (AI05)
│   └── 4 subclasses (AI05 unchanged)
├── GroupStoreError                    (AI06 NEW)
│   ├── GroupReadError
│   ├── GroupWriteError
│   ├── GroupValidationError
│   └── GroupNotFoundError
└── ProjectionError                    (AI04, raised by GraphProjector)
```

All callers `try/except GraphSyncError` continue to catch AI06 errors without modification.

### 5.2 Fatal vs non-fatal

| Scenario | Category | Behaviour |
|---|---|---|
| `us_table.node_uuid` column missing | fatal | `ProjectionError` blocks export, message guides user to Phase 1 migration |
| `groups_{sito}.graphml` corrupt | non-fatal | `GroupReadError` caught in `_merge_groups`, log warning, fall back to SQL-only groups |
| `add_group(name="")` | fatal local | `GroupValidationError` raised in dialog/CLI, not propagated |
| `add_us_to_group` with non-existent us_uuid | non-fatal | warning logged, accepted (no SQL existence check) |
| ad-hoc / SQL name collision | non-fatal | warning logged, "SQL wins" (ad-hoc skipped) |
| `groups=[]` (default) | expected | Stage 4e is a no-op, AC-2 baseline preserved |
| Group with 0 members | non-fatal | inject skipped, info logged |
| `_inject_group_folders` lxml parse error | non-fatal | log error, `add_warning` on graph, file remains structurally valid (but without group folders) |
| Atomic write half-failure | fatal local | `GroupWriteError`, original file untouched, `.tmp` cleaned |
| Import: SQL UPDATE failure | fatal with rollback | `GraphSyncError`, full rollback, `applied=0` |

### 5.3 UI surfacing

- Export dialog: errors → `QMessageBox.critical(type, str(e))` + traceback to Python console (AI05 pattern); warnings → `QgsMessageLog` + `ExportResult.warnings` shown in dialog after export.
- Manage Paradata dialog (Groups tab): `QMessageBox.critical/warning`, plus `[ParadataManager-Groups]` print prefix to Python console.
- Import dialog: when groups exist in GraphML and flag is unchecked, show info banner "N group memberships ignored (enable 'Update SQL on import' to apply)".

### 5.4 CLI exit codes

- `0` — success
- `1` — any `GraphSyncError` (incl. `GroupStoreError`, `ProjectionError`, `ParadataStoreError`)
- `2` — argparse error

### 5.5 Logging

- Print prefixes for live diagnostics: `[GroupStore]`, `[GroupProjector]`, `[GroupInject]`, `[ParadataManager-Groups]` (consistent with AI05 prefixes; useful for remote support).
- `logging.getLogger("pyarchinit.s3dgraphy.sync.groups")` for persistent warnings, routed through QgsMessageLog.
- `graph.add_warning(...)` for export-scoped warnings, surfaced in `ExportResult.warnings`.

## 10. Risks

### Technical

| Risk | Sev | Mitigation |
|---|---|---|
| `_inject_group_folders` breaks the swimlane TableNode when re-parenting US | 🔴 high | Dedicated L4 test verifies output structurally. AC-2 baseline on `groups=[]` is the absolute safety net. Pattern proven by AI05 `_inject_isolated_paradata_nodes`. |
| UUID collision between SQL-derived and ad-hoc groups | 🟡 med | SQL-derived UUID5 deterministic from `(sito, group_kind, name)`; ad-hoc UUID7 random at creation. Real collision impossible unless user names ad-hoc group identically to SQL-derived → "SQL wins" with warning. |
| `GraphMLExporter` mutates `graph.nodes` during export → groups dropped before Stage 4e | 🟡 med | Pre-export snapshot pattern from AI05 reused here. Same bug discovered + fixed during AI05 H.4. |
| Geometry computation overflow with US sparse over many epochs | 🟡 med | E2E test on 12-epoch fixture. yEd renders large dashed bbox correctly. Min margin 20px. |
| Round-trip flag-on: user accidentally moves US to wrong group → SQL corrupted | 🟡 med | Default OFF (D5-C). Explicit checkbox + dialog warning. Atomic transaction with ROLLBACK on any error (AI04 contract). Backup-suggesting message on error. |
| `dimensions_with_data` performance on 100k+ US DB | 🟢 low | Indexed GROUP BY queries. Cached via `db_manager._local_cache` if needed. Target <100ms. Lazy-loaded on dialog open. |

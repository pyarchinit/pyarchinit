integration. It is updated continuously as work progresses on the
`Stratigraph_00001` branch of the PyArchInit plugin and is intended to keep

**Predecessor:** `pg-uuidfix-5.7.8.1-alpha` (`84d625fc`)

### What shipped

Closes the PostgreSQL graphml export gap left by PG-B's incomplete
migration of `populate_graph()` (line 165 `Path(db_path)` + line 190
upstream SQLite-only `PyArchInitImporter`). After PG-Bv2, PG users
get a complete `Extended_Matrix_<site>.graphml` from the matrix
viewer, equivalent to SQLite output.

- **NEW** `modules/s3dgraphy/sync/pyarchinit_pg_importer.py`:
  `import_from_pg(handle, sito, mapping_name)` reads `us_table` via
  SQLAlchemy and builds `StratigraphicNode` + `PropertyNode` objects
  per the `pyarchinit_us_mapping.json` schema. ~120 LOC including
  defensive try/except + cached mapping JSON load.
- **MODIFIED** `graph_projector.py:populate_graph()`: routes through
  `_resolve_db_handle(db_path)` then branches on `handle.is_postgres`.
  PG → new reader; SQLite → upstream PyArchInitImporter (unchanged
  for AC-2 byte-identical baseline). 5 downstream helpers already
  accept DbHandle since PG-B.
- **MODIFIED** `s3dgraphy_dot_bridge.py`: re-removed the
  `if db_path is None: skip` envelope restored by PG-UIFix revert
  (`bc90c86c`). Safe now because populate_graph supports PG.
- **INVERTED** `test_graphml_export_pg_skip_branch_is_present_pending_pg_bv2`
  → `test_graphml_export_runs_on_pg_backend` in
  `tests/sync/test_pg_uifix.py`.
- **NEW** `tests/sync/test_pg_bv2_pg_importer.py` with 2 L0 + 2 L2 PG
  tests.

### Architectural decision: α-narrow

User explicitly chose α-narrow during 2026-05-12 brainstorm:
- α = SQLAlchemy native reader (not temp-SQLite roundtrip or upstream patch)
- narrow = only PG branch uses new reader, SQLite stays on upstream

Rationale: preserves AC-2 byte-identical baseline (critical Phase 2/3
gate), matches PG-B's per-site flip pattern, sets clean predecessor
for future `pg-bv3-unify-importer` milestone that may promote α-wide.

### Bug 2 deferral — populate_graph() SQLite-only constraint

The original PG-UIFix Bug 2 fix (commit `de0174f0`) removed the
`if db_path is None: skip` branch in
`s3dgraphy_dot_bridge.py:191-206` under the premise that PG-B
(5.7.1-alpha, 2026-05-10) had shipped graphml export on PG.
User testing on raw PG (2026-05-12, post initial PG-UIFix
tagging) immediately surfaced a `TypeError`:

    GraphML import failed: TypeError: expected str, bytes or
    os.PathLike object, not Pyarchinit_db_management

**Root cause** (investigation, commit `bc90c86c`):

PG-B (commit `3a2597ab`) did flip 5 SQL helpers in
`modules/s3dgraphy/sync/graph_projector.py` to use
`_resolve_db_handle` (sites at lines 303, 367, 509, 761, 877),
but it missed the orchestrator `populate_graph()` itself which
still has:

- line 165: `db_path = Path(db_path)` — unconditional, throws
  `TypeError` when called with `Pyarchinit_db_management`
  instance
- line 190: `PyArchInitImporter(filepath=str(db_path), ...)` —
  the upstream importer in
  `ext_libs/s3dgraphy/importer/pyarchinit_importer.py` uses
  `sqlite3` directly and accepts only filesystem paths

PG-D L2 tests (`test_paradata_store_pg.py` +
`test_group_store_pg.py`) cover paradata/group store on PG
backend, but **the export pipeline was never tested
end-to-end on raw PG**. That's why the gap survived PG-B
sign-off.

**Decision**: revert the Bug 2 export fix and open a dedicated
**PG-Bv2 milestone** as follow-up. Proper Bug 2 fix needs
~80 LOC (either replace the upstream `PyArchInitImporter`
call with a SQLAlchemy-based us_table reader that constructs
`StratigraphicNode` objects manually, or a temp-SQLite
roundtrip wrapper) — not scope creep into a hotfix milestone.

The restored skip-branch ships a clearer message than the
original `AI04` placeholder:

    GraphML export on PostgreSQL backend is not yet supported:
    pending PG-Bv2 milestone (upstream PyArchInitImporter is
    SQLite-only). DOT and JSON exports are unaffected.

DOT and JSON exports continue to work on both backends.

## yE-Closure — rollout sign-off + tutorial 36 IT/EN extension (5.8.3-alpha)

**Tag:** `yed-import-closure-5.8.3-alpha`
**Date:** 2026-05-14
**Spec:** none (closure-only milestone — no production code)
**Plan:** inline in CHANGELOG
**Predecessor:** `pg-pottery-typefix-5.8.2.3-alpha` (`3f30d368`)

### Sign-off summary

After 6 development milestones (yE-A foundation 2026-05-12 → yE-Closure
2026-05-14, three calendar days), the **yEd-aware graphml import**
rollout is complete. Pyarchinit now imports yEd-authored graphmls
end-to-end via a dedicated dispatch that:

1. Detects flavour (`yEd-raw` vs `pyarchinit-projected`) and routes
   accordingly. Legacy `_run()` path is **byte-identical** for the
   pyarchinit-projected branch (AC-2 enforced across all 6 milestones).
2. Classifies leaves by label prefix into 13 `ClassificationKind`
   destinations.
3. Parses TableNode rows as `PeriodCandidate` and group folders as
   `FolderCandidate` (with auto-derived dimension from label prefix:
   VA→attivita / AR→area / SR→struttura / ST→settore / AM→ambient /
   SG→saggio / QP→quad_par).
4. Lets the user override defaults via a 5-page Qt wizard
   (`YedImportDialog`): classifier / periods / folders / rapporti
   policy / preview. Overrides persist in a sidecar JSON next to the
   graphml.
5. Writes us_table + inventario_materiali_table +
   periodizzazione_table + ParadataStore in a single atomic
   transaction (rollback on any error; dry_run sentinel mirrors
   PG-C). Works on SQLite **and** PostgreSQL via DbHandle.

## yE-E Dialog UX — QWizard 5-page user-overrides (5.8.2-alpha)

**Tag:** `yed-import-dialog-5.8.2-alpha`
**Date:** 2026-05-13
**Spec:** `docs/superpowers/specs/2026-05-13-yed-import-dialog-design.md`
**Plan:** `docs/superpowers/plans/2026-05-13-yed-import-dialog.md`
**Predecessor:** `s3dgraphy-bump-5.8.1-alpha` (`7f5f82a8`)

### What shipped

A 5-page `QWizard` opens automatically when an interactive QGIS
session imports a yEd-raw graphml. Pages: classifier (per-leaf
combobox + bulk reset) / periods (editable periodo+fase cells) /
folders (dimension combobox + value text + `skip` sentinel) /
rapporti policy (4 radio buttons SKIP/FAN_OUT/REPRESENTATIVE/SYNTHETIC) /
preview (dry-run counts).

`YedOverrides` dataclass captures the diffs (only modified fields).
`apply_overrides_to_drafts(drafts, overrides)` is a pure function
that merges them into the yE-A/B/C parser outputs. `import_yed_raw()`
gains an `overrides: YedOverrides | None = None` parameter; `None`
keeps the yE-D hardcoded-defaults behaviour (AC-1 preserved).

Sidecar JSON `<graphml>.yed_overrides.json` persists overrides on
Finish and pre-populates the wizard on re-open. Schema versioned
(`version: 1`). Unknown ClassificationKind values skipped silently
(forward-compat).

Branch hook in `graph_ingestor.py` probes `QApplication.instance()`:
when alive, opens the wizard; when absent (CLI, tests, headless),
falls through to defaults. Defensive try/except on `qgis.PyQt` +
`pyarchinit.gui.yed_import_dialog` imports.

CLI `scripts/import_yed_graphml.py` gains `--overrides PATH` flag
for scripted reruns (`-Recommended in CI`).

# Group B — VocabProvider (Option B per-tool JSON parsing)

**Date:** 6–7 May 2026 · **Status:** Done · **Commits:** 11 (`02d59b07` → `88e70498`)

## What was built

The `modules/s3dgraphy/sync/` package was created from scratch. It is the
PyArchInit-side adapter that consumes s3Dgraphy's three JSON pillars
directly, per the Option B preference declared in Reference Document v0.1
§4.5 (per-tool parsing rather than wrapping a typed s3Dgraphy Python API).
Every consuming surface in PyArchInit — combo boxes, validation hints,
visual rendering — will read from this single provider in subsequent
Groups; the pure-Python core can be unit-tested without QGIS, while the
Qt-aware wrapper integrates with the QGIS event loop only when needed.

The package layout:

| File | Responsibility |
|---|---|
| `modules/s3dgraphy/sync/__init__.py` | Public API surface; re-exports the typed dataclasses and the pure-Python core; provides a lazy accessor `get_vocab_provider()` for the Qt wrapper so non-Qt callers (tests, CLI scripts) never pull in PyQt at import time. |
| `modules/s3dgraphy/sync/vocab_types.py` | Five frozen dataclasses (`UnitType`, `EdgeType`, `VisualRule`, `ParadataType`, `VocabularyVersion`) plus a `Family(str, Enum)` whose values mirror the JSON top-level keys (`stratigraphic_nodes`, `paradata_nodes`, `group_nodes`, …). |
| `modules/s3dgraphy/sync/vocab_provider_core.py` | `VocabProviderCore` — pure-Python loader/queryer. Reads the three pillars from `bundled_dir`, optionally merges overrides from `overrides_dir` per top-level key, and exposes the query API: `get_unit_types(family=...)`, `get_edge_types()`, `get_legal_targets_for(...)`, `get_paradata_types()`, `get_visual_rule(...)`, `get_cidoc_mapping(...)`, `versions` property, optional `min_versions={...}` enforcement at construction time. Accepts the legacy 0.1.30 trailing-space filename `s3Dgraphy_node_datamodel .json` as a fallback. |
| `modules/s3dgraphy/sync/vocab_provider.py` | `VocabProvider` — thin Qt wrapper. Subclasses `QObject` when QGIS is importable, falls back to plain `object` otherwise. Adds a `vocabulary_changed` `pyqtSignal` and a `QFileSystemWatcher` covering both the bundled JSON directory and the user override directory, so combo-box dialogs can subscribe to changes and refresh on the fly. Defaults the bundled directory to `<plugin>/ext_libs/s3dgraphy/JSON_config/` and the overrides directory to `~/.config/pyarchinit/vocab_overrides/`. |

# Group C — i18n adapter refactor

**Date:** 7 May 2026 · **Status:** Done · **Commits:** `69dd3ac9`, `81f71a65`

## What was built

`modules/utility/pyarchinit_i18n_stratigraphic.py`, the long-standing
module that ten different files in the plugin import for unit-type
abbreviations and stratigraphic helpers, was refactored into a
**deprecation shim** over `VocabProvider` while preserving its public
surface byte-for-byte.

The behavioural change is at one place: the hard-coded
`_COMMON_ITEMS = ('USVA','USVB','USVC','USD','CON','VSF','SF','SUS',
'Combinar','Extractor','DOC','property')` tuple is replaced by a lazy
`tuple` subclass (`_CommonItemsProxy`) that resolves on first access by
calling `VocabProvider.get_unit_types()`, filtering out the per-language
US/USM equivalents (which `get_unit_type_items()` adds explicitly), and
appending the legacy non-stratigraphic UI items that are not in the
s3Dgraphy JSON datamodel (`CON`, `SUS`, `Combinar`, `Extractor`, `DOC`,
`property`). When `ext_libs/s3dgraphy/` is missing — for example on a
fresh clone before `modules_installer.py` has run — the proxy falls
back to the legacy 12-item tuple so importers never break.

`UNIT_TYPE_ABBREV` (the per-language US/USM mapping) stays hard-coded:
those abbreviations are pyarchinit-specific UI conventions that are
**not** part of the s3Dgraphy formalism and have no representation in
the JSON datamodel. `_UNIT_TYPE_LABELS`, `STRATIGRAPHIC_RELATIONS` and
the rest of the file below `is_any_unit_prefix` are untouched.

A one-time deprecation log line (via `_warn_deprecation_once()`) fires
the first time the module is imported in a session, pointing new
callers at `modules.s3dgraphy.sync.VocabProvider`. Removal of the
legacy fallback is scheduled for the 6.0.0 release.

## What was tested

- **5 backwards-compatibility regression tests** were added in
  `tests/sync/test_i18n_compat.py` *before* the refactor and run green
  against the legacy implementation, then again green against the new
  adapter — locking the public surface for at least one release cycle.
  Imports verified: `_COMMON_ITEMS`, `UNIT_TYPE_ABBREV`,
  `get_unit_type_items`, `is_us_type`, `ALL_US_ABBREVS`.
- Total `tests/sync/` test count: **24 passed in 0.31 s**.
- A controller-level smoke confirmed the end-to-end behaviour change:
  the unit-type picker dialog (Italian locale) now exposes **27 items**
  (`US, USM, USVs, USVn, SF, VSF, serSU, serUSD, serUSVn, serUSVs, USD,
  TSU, UL, BR, USN, PROP, DOC, EXT, COMB, ANG, PNG, TBNG, CON, SUS,
  Combinar, Extractor, property`) instead of the previous 14, including
  the EM 1.5 types `USVs`, `USVn`, `serSU`, `WorkingUnit (UL)`,
  `NegativeStratigraphicUnit (USN)` and the EM 1.6-bound additions that
  travelled with the 0.1.40 vendor swap. This is the observable value
  delivery of action item AI02 — the unit-type list is now driven by

## Decisions taken

- **`_COMMON_ITEMS` is now a `tuple` subclass, not a plain tuple.**
  `_CommonItemsProxy(tuple)` keeps `isinstance(_COMMON_ITEMS, tuple) is
  True`, supports `+`/`in`/iteration/indexing/equality, and resolves
  lazily on first access. No consumer pickles `_COMMON_ITEMS`, so the
  proxy does not implement `__reduce__`; if a future caller serialises
  it, the resolution can be forced beforehand.
- **The 6 legacy non-stratigraphic items** (`CON`, `SUS`, `Combinar`,
  `Extractor`, `DOC`, `property`) are preserved at the tail of the
  resolved tuple. They are pyarchinit-specific UI categories that are
  not s3Dgraphy node types per Reference Document v0.1 §4.2 — keeping
  them on the pyarchinit side rather than pushing them upstream as
  JSON entries. If they later need to participate in the property
  graph, that becomes a JSON pull request against
  `s3Dgraphy_node_datamodel.json`.
- **`USVA`, `USVB`, `USVC` no longer appear in the picker** as of this
  refactor. Existing database rows that still carry those values keep
  rendering correctly (the `unita_tipo` column is free-text), but new

  one-shot migration that rewrites historical rows; until that
  migration runs, mixed-vocabulary databases are supported by both

# Group E — `node_uuid TEXT` column + UUID v7 backfill migration

**Date:** 7 May 2026 · **Status:** Done · **Commits:** `b63927ae`, `0b93b3aa`, `602d7220`

## What was built

The second one-shot SQL migration of Phase 1: every record in the three
primary stratigraphic tables gets a stable opaque identifier so that, in
later phases, the s3Dgraphy bridge can use it as the `EMid` foreign key
on the property-graph side. Per Reference Document v0.1 §3 *"Identity is
already aligned"*, the PyArchInit per-record UID and the s3Dgraphy `EMid`
will be the same value with two names; this migration creates the value.

| File | Responsibility |
|---|---|
| `modules/s3dgraphy/sync/uuid7.py` | Local UUID v7 generator (RFC 9562 §5.7). Monotonic per-process via a threading lock; when two calls fall within the same millisecond, the second increments the random portion to keep ordering. Will be replaced with `uuid.uuid7()` once the project minimum bumps to Python 3.14+. |
| `scripts/migrations/_2026_05_node_uuid_backfill_lib.py` | Pure-logic library. `TABLES` lists the three target tables (`us_table`, `inventario_materiali_table`, `periodizzazione_table`). `add_columns(db)` adds `node_uuid TEXT` if missing and creates a partial unique index `WHERE node_uuid IS NOT NULL` on each table — the partial predicate lets multiple NULL rows coexist during a partial-failure recovery. `backfill_uuids(db)` assigns `uuid7()` to every NULL `node_uuid`; returns a per-table count of rows updated. |
| `scripts/migrations/2026_05_node_uuid_backfill.py` | CLI front-end. `--dry-run` reports either "needs ALTER TABLE + backfill (no col)" or "N row(s) need backfill" per table without mutating. `--apply` runs `auto_backup_sqlite` + `add_columns` + `backfill_uuids`. `--rollback` restores an explicit backup path. |
| `pyarchinitPlugin.py` (modified) | Second entry in `_init_migrations_menu()`: file picker → confirmation dialog showing the three target tables → `auto_backup_sqlite` + `add_columns` + `backfill_uuids` → result dialog with backup path + per-table counts. Same idempotency guard as the vocab-alignment entry. |

## What was tested

- **5 unit tests** in `tests/sync/test_uuid7.py` cover the generator:
  returns a `uuid.UUID` instance, version field is 7, variant is RFC
  4122 (10xx in byte 8), 1024 successive UUIDs are monotonically
  ordered, the embedded timestamp falls within ±1 ms of wall clock.
- **4 unit tests** in `tests/migrations/test_node_uuid_backfill.py`
  cover the migration: column added on each of the three tables, a
  unique index named `node_uuid` survives the run, a second
  `add_columns` run is a no-op (column count stays 1), backfill assigns
  UUID v7 (canonical 36-char form, version 7 character at index 14,
  pattern `xxxxxxxx-xxxx-7xxx-yxxx-xxxxxxxxxxxx`), and a second
  `backfill_uuids` run returns counts of zero.
- **End-to-end dry-run on the real Volterra database**: the CLI reports
  `needs ALTER TABLE + backfill (no col)` for all three target tables —
  none currently carry a `node_uuid` column. The migration is safe to
  run; user has yet to authorise `--apply`.
- **Combined sync + migrations test run**: `pytest tests/sync/
  tests/migrations/ -q` → **41 passed in 0.38 s** (29 sync + 12
  migrations). Test count breakdown: 5 vocab_types + 14
  vocab_provider_core + 2 vocab_provider_smoke + 5 i18n compat + 5
  uuid7 + 3 migration common + 5 us_vocabulary_alignment + 4
  node_uuid_backfill = 41.

## Decisions taken

- **Local UUID v7 instead of a third-party dependency.** Python's
  stdlib `uuid` module gains `uuid7()` only at 3.14 (currently
  unreleased at the time of this work); the project still has to
  support QGIS Python 3.9+. A 30-line in-repo implementation is
  cheaper than adding a `uuid7` package to `requirements.txt` and
  routing it through the runtime installer. The public API is just
  `uuid7() -> uuid.UUID` so the swap to stdlib later is a one-line
  import change.
- **Partial unique index instead of plain unique.** The index is
  declared `CREATE UNIQUE INDEX … WHERE node_uuid IS NOT NULL`. This
  matters during `--apply` recovery scenarios: if `add_columns`
  succeeded but `backfill_uuids` failed midway, the partially
  populated table has many NULL `node_uuid` rows — a plain unique
  index would reject those because a single NULL is fine but two
  NULLs are not under the SQLite default. The partial predicate
  excludes NULLs from the uniqueness check, so a re-run picks up
  where the previous one left off without dropping the index.
- **Three target tables only.** `tomba_table` is intentionally
  excluded per spec §4.5 last paragraph — burial records are out of
  scope for the bridge in Phase 1. `site_table` is also excluded
  because site-level metadata maps to a different family of s3Dgraphy
  nodes (handled at the property level, not as a typed graph node).
  These exclusions are documented in the spec and will be revisited
  if Phase 2 ingestion proves they need their own UUIDs.
- **No PostgreSQL path in Phase 1.** SQLite-only for now. The same
  logic will move to SQLAlchemy `text()` in Phase 2 when
  `GraphProjector` lands and starts consuming the `node_uuid` column
  in real reads. The CLI already takes a `--db` path so a future
  PostgreSQL DSN form (`postgresql://…`) plugs in cleanly.

## Operational notes

- **Volterra database** has none of the three target columns yet. The
  `--apply` will create all three columns, the partial unique indexes,
  and assign UUID v7 to every existing row. With ~1500 US rows and
  several hundred inventario_materiali / periodizzazione rows, the
  whole migration takes a fraction of a second.
- **Rollback strategy** is the same as Group D: the auto-backup is
  named `<db>.pre_node_uuid_backfill_<UTC>` and lives next to the
  source DB. Restoring is `--rollback <backup_path>`. Combined with
  Group D's vocab-alignment migration, a user can chain the two
  rollbacks if needed (latest backup first, reverse-chronological).

## In-flight Phase 1 hotfix (commit `09b9fbde`)

While preparing the manual smoke gate, the user reported that the
"Extended Matrix" export produced an empty `.graphml` file alongside
the (correct) DOT and `_s3dgraphy.json` outputs. Investigation traced
the root cause to a long-standing bug in
`resources/dbfiles/dot.py::Edge.initFromString()`:

- The DOT file emitted by `S3DGraphyDotBridge` uses node IDs that
  include the site name, e.g. `"Castelseprio Casa Piccoli_1_36"`.
  These IDs are properly double-quoted in the DOT source.
- The edge parser split the edge line via
  `line.replace('->', ' ').split()` — naive whitespace split that did
  not respect double-quoted runs.
- Result: every edge ended up with `src="Castelseprio"`,
  `dest="Casa"`. Both lookups missed the (correctly-named) node
  dictionary keys, so `dottoxml.exportGraphml` wrote zero `<edge>`
  elements and, with this dataset, produced a 0-byte file.

The bug is **pre-existing** (unrelated to the Phase 1 refactor — git
blame on the affected line predates this branch). Phase 2 / AI03 was
already going to retire the entire DOT-parser code path by delegating
GraphML serialisation to s3Dgraphy's always-EM-current writer; that
remains the proper long-term fix. In the meantime this hotfix
unblocks the Phase 1 smoke gate so the user can validate the new
combo-box behaviour against a working Extended Matrix export.

The hotfix is a 36-line addition to `dot.py`: a new
`_split_respecting_quotes()` helper alongside the existing
`findUnquoted()` / `findLastUnquoted()` siblings, and a single-line
swap at the call site in `Edge.initFromString()`. Regression-tested
against four cases (bare identifiers, single-token quoted,
multi-word quoted, multi-word with attributes); on the real
Volterra/Castelseprio export the parser now extracts 655 edges with
correct full src/dest (was: 655 with truncated src/dest, all of which
fell through the node-lookup and were lost during GraphML write).

The 42-test sync + migrations suite is unaffected (the fixed code
path is in legacy DOT/GraphML production code that has no
unit-test coverage today — this gap will close in Phase 2 when the
delegation to s3Dgraphy lands and replaces the parser entirely).

## Second in-flight Phase 1 hotfix (commit `d0dcd828`)

The first hotfix patched `resources/dbfiles/dot.py`, but the user
reported that the plugin loads `dot.py` from `~/pyarchinit/bin/`
rather than directly from the plugin tree. Investigation surfaced
the deployment mechanism:

- `pyarchinit_Folder_installation.install_dir()` copies a list of
  utility files (`dot.py`, `dottoxml.py`, `X11Colors.py`,
  `EM_palette.graphml`, etc.) from `resources/dbfiles/` into
  `~/pyarchinit/bin/` at first install.
- This routine runs **only** when `~/pyarchinit/` does not exist
  (`__init__.py:805`: `if not os.path.exists(PYARCHINIT_HOME):
  fi.install_dir()`). On every subsequent QGIS reload the
  install path is skipped and the runtime copies stay frozen at
  whatever version shipped on the day of first install.
- `Pyarchinit_OS_Utility.copy_file()` is itself a no-op when the
  destination already exists (`if os.access(destination, 0):
  return 0`), so even if the install routine were re-run it
  wouldn't refresh anything.

Net effect: a bug fix in `resources/dbfiles/dot.py` never reaches
the actual `~/pyarchinit/bin/dot.py` that the plugin imports at
runtime. Users have a stale copy and keep hitting the original bug.

The fix introduces an auto-refresh path:

- A new class constant `MAINTENANCE_BIN_FILES` on
  `pyarchinit_Folder_installation` lists the four files the plugin
  owns end-to-end (`dot.py`, `dottoxml.py`, `X11Colors.py`,
  `EM_palette.graphml`). User-editable assets (`epoche_storiche.csv`,
  `thesaurus_template.csv`, `template_report_adarte.docx`) are
  deliberately excluded so their first-install copy is preserved.
- A new method `install_or_update_maintenance_files()` compares
  mtime + size between the bundled source under `resources/dbfiles/`
  and the installed copy under `~/pyarchinit/bin/`, overwriting
  via `shutil.copy` (force-overwrite) when src is strictly newer
  or sizes differ. Idempotent on re-runs.
- `__init__.py` calls the method on every plugin reload (in the
  branch where `~/pyarchinit/` already exists), wrapped in
  try/except so a refresh failure never blocks plugin startup.
- The same method is also called at the end of `installConfigFile()`
  so first installs go through the same code path.

End-to-end tested locally: a stale 1000-byte dot.py was refreshed
to the bundled 49920-byte version on first call; the second call
was a no-op (mtime + size match). The user's real
`~/pyarchinit/bin/dot.py` was also refreshed in the same run, so
the Extended Matrix smoke test can proceed without a manual delete
of the old file.

# Phase 2 — AI03 closed (GraphML delegation)

**Date:** 7 May 2026 · **Status:** **Done.** · **Commits:** approx
`8df46836..HEAD` (the pre-AI03 documentation commit was `8df46836`;
the AI03 implementation opens at `19924971` with the `ExportResult`
dataclass and closes with this group's metadata bump and changelog).

This phase delivered AI03 of the T5.4 backlog: the GraphML branch of
the "Extended Matrix" export now delegates to s3Dgraphy's
`PyArchInitImporter` + `GraphMLExporter` instead of the legacy
DOT→GraphML pipeline. The cut-over is clean: the legacy code path is
deleted, not gated behind a fallback flag. The four Extended Matrix
limitations carried over from Phase 1 (empty graphml when the
spatial-grouping flag was set, no period swimlanes, partial edge
styling, no transitive reduction) close in a single move, because
the s3Dgraphy exporter handles all four natively.

## What was built

A new orchestration module, `modules/s3dgraphy/sync/graphml_writer.py`,
hosts three pure helpers — `build_pyarchinit_mapping()`,
`count_epoch_nodes()`, `count_topological_edges()` — and a single
public entry point `write_graphml_via_s3dgraphy(sqlite_path,
output_path) -> ExportResult`. The orchestrator opens the SQLite via
SQLAlchemy, constructs `PyArchInitImporter(connection_url=...,
mapping_name="pyarchinit_us_mapping", overwrite=True)`, populates
the in-memory graph, and then runs an internal helper
`_enrich_pyarchinit_graph()` which adds the artefacts that the
importer does not produce on its own: `EpochNode`s built from
`periodizzazione_table`, `has_first_epoch` edges from each US to its
period, and topological `is_before` edges parsed from
`us_table.rapporti`. With the graph fully decorated, the orchestrator
calls `GraphMLExporter(graph, output_path).export()` (which performs
its own transitive reduction) and returns an `ExportResult` carrying
`node_count`, `edge_count`, `epoch_count`, `tred_removed_edges` and
the optional warnings list. `tred_removed_edges` is computed
arithmetically as `pre_edges - post_edges` rather than parsed from
the GraphML output: this avoids a fragile regex over the exporter's
implementation details. The mapping name is `pyarchinit_us_mapping`
(the implementation plan's earlier `pyarchinit` was wrong — it does
not match the JSON file shipped with s3Dgraphy 0.1.40).

The bridge cut-over was performed in five per-file commits so any
single step is easy to revert. `s3dgraphy_dot_bridge.py` had its
`_convert_dot_to_graphml()` private function and its `dottoxml`
import removed, and its dispatcher now calls the new orchestrator
when the requested format is GraphML; the DOT branch is left
untouched because Harris-matrix exporters still rely on it.
`graphml_spatial_enhancer.py` was deleted in full — it was a
post-processor that nested nodes inside `y:ProxyAutoBoundsNode`
subgraphs after the GraphML had already been written, and
`GraphMLExporter` emits those swimlanes natively. `spatial_grouping_manager.py`
lost both its `apply_grouping_to_dot()` function and the
`SpatialGroupingDialog` Qt class (the latter was tied to the old DOT
pipeline); the module now keeps only the grouping data structures
that are still consulted elsewhere. `S3DGraphyExportDialog` lost the
`cb_spatial_grouping` checkbox plus all the code that read it, since
spatial grouping is now driven by the s3Dgraphy exporter without a
user toggle. Finally, `resources/dbfiles/dot.py` and `dottoxml.py`
were intentionally left in place: the legacy Harris-matrix exporters
still consume them and they are out of AI03 scope (revisit in AI04).

The `pyarchinit_db_manager` module gained a new `get_sqlite_path()`
method that returns the absolute path of the SpatiaLite file when
the active backend is SQLite, and `None` otherwise (PostgreSQL,
unconfigured manager). The bridge consults this signal before
invoking the orchestrator and skips the GraphML branch with an Info
dialog ("PostgreSQL backend will be supported in AI04") when the
project is wired to PostgreSQL/PostGIS. This is the agreed graceful
degradation for AI03 — `PyArchInitImporter` accepts only a SQLite
`connection_url` today, and AI04 will widen its surface.

The export dialog summary was rewritten to surface the per-format
metrics returned by the new orchestrator. After every Extended
Matrix export, the user now sees, for each emitted format
(GraphML/DOT/JSON), the node, edge and epoch counts, the number of
transitive edges removed, and any warnings (e.g. a notice when
`periodizzazione_table` is empty and `epoch_count` falls back to
zero). This replaces the pre-AI03 binary "Export completed" message.

## What was tested

The test suite under `tests/sync/` grew by 16 tests (total: 59
passing). Three of them are unit tests on the pure helpers
(`build_pyarchinit_mapping`, `count_epoch_nodes`,
`count_topological_edges`). Four exercise the end-to-end orchestrator
against a deterministic fixture, `tests/sync/fixtures/mini_volterra.sqlite`,
and are mapped one-to-one to the four Extended Matrix limitations:
the GraphML output is non-empty when grouping is active, period
swimlanes are present in the output, `is_before` edges carry their
intended y:Edge styling, and `tred_removed_edges` is greater than
zero on a graph that contains a redundant transitive edge. Four more
tests cover `db_manager.get_sqlite_path()` (valid SQLite path,
missing SQLite path, PostgreSQL → `None`, uninitialized manager →
`None`), and five pre-existing exception-helper tests round out the
file.

The fixture itself, `mini_volterra.sqlite`, contains six US records
with cross-cutting `rapporti` (so the transitive-reduction case is
non-trivial), three rows in `periodizzazione_table`, and a
simplified geometry layer. It is committed as a binary blob so the
suite runs reproducibly without depending on production DBs. The
`_enrich_pyarchinit_graph()` helper inside the orchestrator is the
deliberate workaround for a Phase-1 finding: `PyArchInitImporter`
alone does not emit `EpochNode`s, `has_first_epoch` edges or
topological `is_before` edges. The orchestrator therefore enriches
the graph before handing it to the exporter; the four pipeline tests
all pass because of this enrichment, and the workaround is
documented in-line in the module so the next maintainer knows why
it exists.

## Decisions taken

The cut-over is **clean — there is no fallback flag**. The legacy
DOT→GraphML pipeline was a maintenance liability (it produced an
empty file under the spatial-grouping flag and could not emit
swimlanes); shipping it side-by-side with the s3Dgraphy delegation
would have doubled the surface area without buying any user value.
Rolling back is, instead, a `git revert` of the five per-file Group E
commits.

`tred_removed_edges` is computed arithmetically (`pre_edges -
post_edges`) rather than parsed from the GraphML output via regex.
The arithmetic version is robust to formatting changes inside
`GraphMLExporter` and is symmetrical to the way `node_count`/`edge_count`
are produced.

The mapping name passed to `PyArchInitImporter` is
`pyarchinit_us_mapping`, not `pyarchinit` as the early plan draft
suggested. The vendored s3Dgraphy 0.1.40 ships
`pyarchinit_us_mapping.json` under `ext_libs/s3dgraphy/`, and this
is the only mapping name the importer recognises out of the box.
This was caught while wiring up the first pipeline test and is
documented in the orchestrator docstring.

`python scripts/modules_installer.py` only if their

metadata bump is documentation-only and adds no new third-party
dependencies. The Phase 2 manual smoke gate (G.4) and tag (G.5) are

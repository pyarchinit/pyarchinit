# Phase 1 (Foundation) — overview

Phase 1 closes action items AI01–AI05 from the T5.4 meeting of 4 May 2026. The
six deliverables are:

1. Vendor s3Dgraphy 0.1.30 → 0.1.40 inside `ext_libs/`.
2. New `VocabProvider` that parses the three s3Dgraphy JSON pillars directly
   (Option B per Reference Document v0.1 §4.5).
3. Refactor `pyarchinit_i18n_stratigraphic.py` as a backwards-compatible
   adapter over `VocabProvider`.
4. One-shot vocabulary alignment migration: `USVA/USVB → USVs`,
   `USVC → USVn`.
5. UUID v7 backfill on the three primary tables (`us_table`,
   `inventario_materiali_table`, `periodizzazione_table`) using a `node_uuid
   TEXT` column with a partial unique index.
6. Refactor 12 files under `modules/s3dgraphy/` so they consult `VocabProvider`
   instead of hardcoded EM 1.4 tables.

Plan structure: 8 Groups (0 + A through G), 37 tasks, ~150 steps,
≈ 30 commits expected. The release that closes Phase 1 is tagged
`phase1-foundation-5.1.0-alpha`. A rollback tag `pre-s3dgraphy-040` was
created at the start of execution; everything is recoverable.

# Group B — VocabProvider (Option B per-tool JSON parsing)

**Date:** 6–7 May 2026 · **Status:** Done · **Commits:** 11 (`02d59b07` → `88e70498`)

## What was built

The `modules/s3dgraphy/sync/` package was created from scratch. It is the
PyArchInit-side adapter that consumes s3Dgraphy's three JSON pillars
directly, per the Option B preference declared in Reference Document v0.1
§4.5 (per-tool parsing rather than wrapping a typed s3Dgraphy Python API).
Every consuming surface in PyArchInit — combo boxes, validation hints,
visual rendering — will read from this single provider in subsequent
Groups; the pure-Python core can be unit-tested without QGIS, while the
Qt-aware wrapper integrates with the QGIS event loop only when needed.

The package layout:

| File | Responsibility |
|---|---|
| `modules/s3dgraphy/sync/__init__.py` | Public API surface; re-exports the typed dataclasses and the pure-Python core; provides a lazy accessor `get_vocab_provider()` for the Qt wrapper so non-Qt callers (tests, CLI scripts) never pull in PyQt at import time. |
| `modules/s3dgraphy/sync/vocab_types.py` | Five frozen dataclasses (`UnitType`, `EdgeType`, `VisualRule`, `ParadataType`, `VocabularyVersion`) plus a `Family(str, Enum)` whose values mirror the JSON top-level keys (`stratigraphic_nodes`, `paradata_nodes`, `group_nodes`, …). |
| `modules/s3dgraphy/sync/vocab_provider_core.py` | `VocabProviderCore` — pure-Python loader/queryer. Reads the three pillars from `bundled_dir`, optionally merges overrides from `overrides_dir` per top-level key, and exposes the query API: `get_unit_types(family=...)`, `get_edge_types()`, `get_legal_targets_for(...)`, `get_paradata_types()`, `get_visual_rule(...)`, `get_cidoc_mapping(...)`, `versions` property, optional `min_versions={...}` enforcement at construction time. Accepts the legacy 0.1.30 trailing-space filename `s3Dgraphy_node_datamodel .json` as a fallback. |
| `modules/s3dgraphy/sync/vocab_provider.py` | `VocabProvider` — thin Qt wrapper. Subclasses `QObject` when QGIS is importable, falls back to plain `object` otherwise. Adds a `vocabulary_changed` `pyqtSignal` and a `QFileSystemWatcher` covering both the bundled JSON directory and the user override directory, so combo-box dialogs can subscribe to changes and refresh on the fly. Defaults the bundled directory to `<plugin>/ext_libs/s3dgraphy/JSON_config/` and the overrides directory to `~/.config/pyarchinit/vocab_overrides/`. |

## What was tested

The TDD discipline laid out in plan §B was followed strictly. Every method
on `VocabProviderCore` was preceded by a failing test, then implemented,
then verified green before commit. The final test count under
`tests/sync/`:

- **17 fixture-driven tests** in `test_vocab_types.py` and
  `test_vocab_provider_core.py` covering typed dataclass shape, JSON
  loading, family filtering, edge-type loading, paradata loading, visual
  rules, CIDOC lookup, version exposure, minimum-version gating (pass +
  raise), legacy trailing-space filename fallback, and override merge
  semantics (a partial paradata override must not erase bundled
  stratigraphic types).
- **2 smoke tests** in `test_vocab_provider_smoke.py` running against the
  real `ext_libs/s3dgraphy/JSON_config/` 0.1.40 vendor installed in
  Group A. The smoke tests confirm `US`, `USVs`, `USVn` appear in the
  stratigraphic family on the actual vendored datamodel and that the
  per-pillar version field is populated. Auto-skipped when `ext_libs/`
  is missing (fresh clone before `modules_installer.py` runs).

Total: **19 passed in 0.20 s**, run via `pytest tests/sync/ -v` from the
plugin root. The runner is configured by a minimal `pyproject.toml`
([`tool.pytest.ini_options`] with `testpaths = ["tests/sync"]` and
`--import-mode=importlib --confcutdir=tests/sync`) so the new suite can
run from the CLI without QGIS bootstrap — needed because the plugin's
top-level `__init__.py` imports `qgis.PyQt.*` at module load.

Two-stage review was applied to the foundation pair (Tasks B.1 + B.2): an
independent spec-compliance reviewer confirmed byte-level match to the
plan literal source, and a code-quality reviewer approved with minor
follow-ups, all four of which were folded into the next subagent batch
without requiring a separate fix commit.

## Decisions taken

- **Pure core / Qt wrapper split** is locked in. The pure-Python
  `VocabProviderCore` is the only surface tests exercise; the Qt
  `VocabProvider` is a thin singleton-with-signal that any QGIS dialog
  can subscribe to without inheriting test infrastructure. This keeps
  the test suite QGIS-free and runnable in CI / from a bare shell.
- **`pyproject.toml` is the canonical pytest config location** for this
  plugin, scoped narrowly to `tests/sync`. Legacy tests under
  `tests/test_*.py` already required QGIS to run, so no change in
  behaviour for them; the new suite is additive.
- **The 0.1.30 trailing-space filename** (`s3Dgraphy_node_datamodel .json`
  with a literal space before the extension) is preserved as a fallback
  in `_NODE_DATAMODEL_NAMES`. It will only be removed when 0.1.30
  support is officially dropped — annotated inline so future
  maintainers find the answer without archaeology.
- **Override priority is per top-level key, not whole-file**, locked in
  by a regression test that builds a partial paradata-only override and
  asserts the bundled stratigraphic types still surface from
  `get_unit_types()`. A datacenter pulling a partial vocabulary update
  will not silently erase locally available types.
- **Public API frozen for Phase 1**: `VocabProviderCore`, `UnitType`,
  `EdgeType`, `VisualRule`, `ParadataType`, `VocabularyVersion`,
  `Family`, `get_vocab_provider()`. `SyncEngine`, `GraphProjector`,
  `ParadataStore`, `ConflictResolver` are deliberately absent — they
  belong to Phases 2 and 3.

## Plan corrections folded in mid-execution

- **`pyproject.toml` was not in the plan** but is required so the new
  test suite can run from a bare CLI. The deviation was approved during
  spec review of B.1 + B.2; the file is 6 lines, single-section, and
  scoped to `tests/sync` so legacy tests are unaffected.
- **`EdgeType.allowed_pairs` annotation tightened** from raw `tuple` to
  `tuple[tuple[str, str], ...]` (review follow-up), preserving the
  hashability contract of frozen dataclasses.

# Group C — i18n adapter refactor

**Date:** 7 May 2026 · **Status:** Done · **Commits:** `69dd3ac9`, `81f71a65`

## What was built

`modules/utility/pyarchinit_i18n_stratigraphic.py`, the long-standing
module that ten different files in the plugin import for unit-type
abbreviations and stratigraphic helpers, was refactored into a
**deprecation shim** over `VocabProvider` while preserving its public
surface byte-for-byte.

The behavioural change is at one place: the hard-coded
`_COMMON_ITEMS = ('USVA','USVB','USVC','USD','CON','VSF','SF','SUS',
'Combinar','Extractor','DOC','property')` tuple is replaced by a lazy
`tuple` subclass (`_CommonItemsProxy`) that resolves on first access by
calling `VocabProvider.get_unit_types()`, filtering out the per-language
US/USM equivalents (which `get_unit_type_items()` adds explicitly), and
appending the legacy non-stratigraphic UI items that are not in the
s3Dgraphy JSON datamodel (`CON`, `SUS`, `Combinar`, `Extractor`, `DOC`,
`property`). When `ext_libs/s3dgraphy/` is missing — for example on a
fresh clone before `modules_installer.py` has run — the proxy falls
back to the legacy 12-item tuple so importers never break.

`UNIT_TYPE_ABBREV` (the per-language US/USM mapping) stays hard-coded:
those abbreviations are pyarchinit-specific UI conventions that are
**not** part of the s3Dgraphy formalism and have no representation in
the JSON datamodel. `_UNIT_TYPE_LABELS`, `STRATIGRAPHIC_RELATIONS` and
the rest of the file below `is_any_unit_prefix` are untouched.

A one-time deprecation log line (via `_warn_deprecation_once()`) fires
the first time the module is imported in a session, pointing new
callers at `modules.s3dgraphy.sync.VocabProvider`. Removal of the
legacy fallback is scheduled for the 6.0.0 release.

## What was tested

- **5 backwards-compatibility regression tests** were added in
  `tests/sync/test_i18n_compat.py` *before* the refactor and run green
  against the legacy implementation, then again green against the new
  adapter — locking the public surface for at least one release cycle.
  Imports verified: `_COMMON_ITEMS`, `UNIT_TYPE_ABBREV`,
  `get_unit_type_items`, `is_us_type`, `ALL_US_ABBREVS`.
- Total `tests/sync/` test count: **24 passed in 0.31 s**.
- A controller-level smoke confirmed the end-to-end behaviour change:
  the unit-type picker dialog (Italian locale) now exposes **27 items**
  (`US, USM, USVs, USVn, SF, VSF, serSU, serUSD, serUSVn, serUSVs, USD,
  TSU, UL, BR, USN, PROP, DOC, EXT, COMB, ANG, PNG, TBNG, CON, SUS,
  Combinar, Extractor, property`) instead of the previous 14, including
  the EM 1.5 types `USVs`, `USVn`, `serSU`, `WorkingUnit (UL)`,
  `NegativeStratigraphicUnit (USN)` and the EM 1.6-bound additions that
  travelled with the 0.1.40 vendor swap. This is the observable value
  delivery of action item AI02 — the unit-type list is now driven by
  s3Dgraphy's JSON datamodel and will pick up future EM revisions
  without code changes in pyarchinit.

## Decisions taken

- **`_COMMON_ITEMS` is now a `tuple` subclass, not a plain tuple.**
  `_CommonItemsProxy(tuple)` keeps `isinstance(_COMMON_ITEMS, tuple) is
  True`, supports `+`/`in`/iteration/indexing/equality, and resolves
  lazily on first access. No consumer pickles `_COMMON_ITEMS`, so the
  proxy does not implement `__reduce__`; if a future caller serialises
  it, the resolution can be forced beforehand.
- **The 6 legacy non-stratigraphic items** (`CON`, `SUS`, `Combinar`,
  `Extractor`, `DOC`, `property`) are preserved at the tail of the
  resolved tuple. They are pyarchinit-specific UI categories that are
  not s3Dgraphy node types per Reference Document v0.1 §4.2 — keeping
  them on the pyarchinit side rather than pushing them upstream as
  JSON entries. If they later need to participate in the property
  graph, that becomes a JSON pull request against
  `s3Dgraphy_node_datamodel.json`.
- **`USVA`, `USVB`, `USVC` no longer appear in the picker** as of this
  refactor. Existing database rows that still carry those values keep
  rendering correctly (the `unita_tipo` column is free-text), but new
  rows will use the canonical `USVs` and `USVn`. Group D ships the
  one-shot migration that rewrites historical rows; until that
  migration runs, mixed-vocabulary databases are supported by both
  layers.

## Downstream consumers (informational)

Ten files import `pyarchinit_i18n_stratigraphic`. The highest-traffic
ones are `tabs/US_USM.py` (the main US/USM controller, where the
picker dialog lives), `modules/db/pyarchinit_db_manager.py`,
`modules/utility/pyarchinit_exp_USsheet_pdf.py`,
`modules/utility/pyarchinit_matrix_exp.py`, and
`tabs/Interactive_matrix.py`. None were modified by this refactor —
the public surface stayed the same. They will start seeing the
expanded vocabulary the next time the plugin reloads.

# Group D — USVA/USVB → USVs, USVC → USVn one-shot migration

**Date:** 7 May 2026 · **Status:** Done · **Commits:** `b8a1f227`, `15efd4da`, `b9a10b04`

## What was built

A one-shot SQL migration tool that aligns historical PyArchInit databases
to the canonical EM 1.5 vocabulary now exposed through `VocabProvider`.
Implementation is split across a shared scaffold and a migration-specific
library/CLI pair so subsequent migrations (Group E and beyond) can reuse
the boilerplate.

| File | Responsibility |
|---|---|
| `scripts/migrations/_common.py` | Shared CLI scaffold: `parse_argv()` providing the standard `--dry-run / --apply / --rollback BACKUP_PATH` action group with required `--db` argument, and `auto_backup_sqlite(db, tag)` which copies the SQLite file to `<db>.pre_<tag>_<UTC>` via `shutil.copy2`. |
| `scripts/migrations/_2026_05_us_vocabulary_alignment_lib.py` | Pure-logic library exposing the `REPLACEMENTS` mapping (`USVA → USVs`, `USVB → USVs`, `USVC → USVn`), `plan_changes(db)` for the read-only count report, and `apply_changes(db)` for the in-place rewrite. Library/CLI split lets tests exercise the logic without spawning a subprocess. |
| `scripts/migrations/2026_05_us_vocabulary_alignment.py` | CLI front-end. On `--apply`, calls `auto_backup_sqlite` first, then `apply_changes`. On `--rollback BACKUP_PATH`, restores the named backup via `shutil.copy2`. |
| `pyarchinitPlugin.py` (modified) | New `_init_migrations_menu()` and `_run_vocab_alignment_migration()` methods wired into `_init_stratigraph_sync()`, called from all 4 language branches of `initGui()`. The user-facing flow: file picker → `plan_changes` → `QMessageBox.question` confirm → `auto_backup_sqlite` + `apply_changes` → result dialog. Idempotent via a `_migrations_menu_wired` guard. Imports use the `qgis.PyQt.QtWidgets` abstraction so the same code runs on Qt5 and Qt6. |

## What was tested

- **3 unit tests** in `tests/migrations/test_common.py` cover the scaffold:
  `auto_backup_sqlite` copies the file and preserves data; `parse_argv`
  accepts each of the three actions; bare `--db` exits with `SystemExit`.
- **5 unit tests** in `tests/migrations/test_us_vocabulary_alignment.py`
  cover the migration: `plan_changes` reports correct counts on a 5-row
  fixture without mutating, `apply_changes` rewrites correctly, the
  second `--apply` is a no-op (idempotency), and the `REPLACEMENTS`
  mapping is locked in.
- **End-to-end dry-run on the real Volterra database**
  (`pyarchinitcs_SCHEDE2024.sqlite`): zero rows have `USVA`/`USVB`/`USVC`
  in `unita_tipo`, so the migration will be a no-op there. The CLI
  exits cleanly without touching the file. Recorded as a safety check
  before the manual smoke gate at G.4.
- **Combined sync + migrations test run**: `pytest tests/sync/
  tests/migrations/ -q` → **32 passed, 1 warning** (the warning is a
  `datetime.utcnow()` deprecation harmless on Python 3.13; deferred for
  cleanup when minimum Python bumps).

## Decisions taken

- **Library/CLI split for every migration.** `_2026_05_us_vocabulary_alignment_lib.py`
  contains the pure logic; `2026_05_us_vocabulary_alignment.py` is the
  thin CLI wrapper. This pattern lets `tests/migrations/` exercise the
  logic directly without `subprocess.run`, and lets the QGIS menu
  handler import the library functions directly without spawning a
  child process. Group E will follow the same split.
- **Idempotency by construction.** `apply_changes` issues `UPDATE
  us_table SET unita_tipo = ? WHERE unita_tipo = ?` for each
  `REPLACEMENTS` pair; the second run finds zero matching rows and
  returns rowcount 0 per source. No "has-already-run" sentinel column
  needed.
- **Auto-backup names live next to the database file.** The naming
  pattern `<db>.pre_<tag>_<YYYYmmddTHHMMSSZ>` keeps backups
  side-by-side with their source DB so users immediately see them in a
  file-picker. `--rollback` takes the explicit backup path so the user
  is in control of which backup gets restored.
- **`.gitignore` whitelist extended.** The same pattern that Group 0
  applied to `tests/sync/` (re-include the new test directory inside
  the existing `/tests/*` blanket ignore) was applied to
  `tests/migrations/`. Both are tracked; legacy `tests/test_*.py`
  files unaffected.
- **`pyarchinitPlugin.py` is the canonical filename** — lowercase `p` —
  matches the existing repository state, not the camelCase variant
  referenced in some older notes. The new menu wiring lives there.

## Operational notes

- **Volterra database (`pyarchinitcs_SCHEDE2024.sqlite`) needs no
  alignment.** The dry-run reports 0 rows in every replacement bucket.
  The migration is therefore a no-op when first applied to it; if the
  user later imports rows from a legacy export that used `USVA`/`USVB`/
  `USVC`, re-running the migration cleans them up.
- **Other historical PyArchInit databases** (e.g. `pyarchinit_db.sqlite`,
  the development scratchpad) should each be dry-run before `--apply`
  to know what will change. The QGIS menu does this automatically: the
  confirmation dialog shows the per-source count before mutating.
